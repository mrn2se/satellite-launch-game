# Satellite Launch Game - Development Agent

## Project Overview

**Satellite Launch Game** is a real-time 3D space mission simulator built with modern web technologies (Vite + TypeScript + Three.js). Players launch a satellite from Earth and navigate it through the solar system using realistic Newtonian physics.

### Core Technologies
- **Runtime**: TypeScript, Three.js, WebGL
- **Build**: Vite
- **Testing**: Vitest
- **Input**: Keyboard, Mouse, Touch, Gamepad
- **License**: Mozilla Public License 2.0 (MPL-2.0)

## Architecture & Design Patterns

### Physics Engine (`src/physics.ts`)
```
Key Functions:
- vec3Add, vec3Sub, vec3Scale, vec3Length, vec3Normalize → Vector math
- gravitationalAcceleration(position, source) → Single body gravity
- totalGravitationalAcceleration(position, bodies[]) → Superposition
- integrateStep(pos, vel, dt, bodies) → Velocity Verlet advancement
- checkCollision(satPos, bodies) → Collision detection
- gravityFieldAt(position, bodies) → Get field vector for visualization
```

### Game Simulation (`src/simulation.ts`)
```
State Machine: setup → running → paused → won/lost
Key Methods:
- launch(angle, speed) → Initialize satellite from Earth
- update(realDt) → Main update loop with sub-stepping
- applyCorrection(maneuver) → Mid-flight course adjustment
- pause/resume() → State transitions
```

### Rendering (`src/renderer.ts`)
```
Three.js Architecture:
- Scene with 8 planet meshes + trails + correction arrow
- Two cameras: orbitCamera (free look) + satelliteCamera (first-person)
- OrbitControls for camera interaction
- Dynamic gravity field arrows (recreated each frame)
- Sprite labels for planet names
```

### Input Management (`src/ui.ts`, `src/gamepad.ts`)
```
Input Stack:
1. Keyboard (arrow keys, space, hotkeys)
2. Mouse (orbit camera, click focus)
3. Touch (joystick drag, floating buttons)
4. Gamepad (all features mapped)
5. Mobile UI (panels, sliders, buttons)
```

## Known Limitations & Technical Debt

### Performance Issues
- **Gravity field rendering**: Creates ~100-300 Arrow3D helpers per frame when enabled
  - **Impact**: 20-30% framerate hit on mid-range GPUs
  - **Solution**: Future GPU compute shader or decimation

### Physics Limitations
- No collision time-stepping (large timesteps may miss collisions)
- Planets treated as point masses (no shape collision)
- No perturbations from minor bodies
- No atmospheric drag or relativistic effects
- Discontinuous deltaV application (unrealistic)

### Gameplay Issues
- Limited educational value (no help system)
- No mission variety (only 5 destinations)
- No difficulty scaling
- No undo/rewind functionality

## Development Workflow

### Setting Up Dev Environment
```bash
pnpm install
pnpm dev          # Start Vite dev server
pnpm test:watch   # Watch tests during development
```

### Making Changes
1. **Physics Changes**: Update `src/physics.ts`, add tests in `tests/trajectory.test.ts`
2. **Visuals**: Modify `src/renderer.ts`, test in dev server
3. **UI/Input**: Update `src/ui.ts` and `src/gamepad.ts`, test on multiple devices
4. **Styles**: Edit `src/style.css`, test responsive breakpoints

### Before Committing
```bash
pnpm build           # Verify TypeScript + Vite build
pnpm test            # Run all tests
pnpm lint            # (future: add eslint)
git diff             # Review changes
```

## Code Quality Standards

- Strict mode enabled (`strict: true`)
- No unused variables (`noUnusedLocals: true`)
- No unused parameters (`noUnusedParameters: true`)

### Style
- Functional components where possible
- Single responsibility principle
- Clear variable naming (physics domain terms)
- Comments for complex logic

### Performance
- Debounce window resize events
- Lazy-load 3D assets
- Object pooling for arrows (future)
- Avoid creating garbage in game loop
